Now we are moving into permitting.

Goal:
Add a Permitting Tracker foundation so contractors can track project permits, applications, approvals, inspections, expiration dates, jurisdictions, permit numbers, and required documents.

Primary contractor story:
A company owner/admin opens a project and can create permits tied to that project. Each permit tracks type, jurisdiction, status, submission date, approval date, expiration date, permit number, inspection dates, notes, client visibility, and whether the permit is blocking work.

Primary client story:
A client may eventually view client-visible permit status updates, but the first version should be careful not to expose internal notes or sensitive company/admin data.

Preferred direction unless the code suggests otherwise:

* Add a dedicated `project_permits` table.
* Keep permits project-scoped and company-scoped.
* Add a Project Detail tab called “Permits.”
* Company Owner and Company Admin can create/edit/delete permits.
* Regular internal contractors can view permits but should not edit unless there is already a safe internal edit pattern.
* Clients should only see permits where `clientVisible = true`, and only safe client-facing fields.
* Subcontractors/notaries should not see permit data in this phase.
* Do not connect permits to timeline/Gantt automatically yet.
* Do not auto-change project status/progress from permit statuses.
* Do not connect permits to budgets, expenses, invoices, selections, daily logs, production sheets, notifications, or emails yet.
* Keep inspections simple in the first version, using fields on the permit row instead of a child inspection table unless the code strongly suggests otherwise.
* Include documentUrl or permitPortalUrl as plain text/link fields if useful.
* Defer file/image uploads unless there is already a very safe upload pattern that can be reused without expanding scope.

Client-safe response rule:
If clients can view permits in Phase 18, the API must shape the client response separately.

Client-safe permit fields may include:

* id
* projectId
* permitType
* title
* description
* jurisdiction
* permitNumber, if safe
* status
* submittedDate
* approvedDate
* issuedDate
* expirationDate
* inspectionDate
* finalInspectionDate
* nextActionDate
* blockingWork

Do not include these fields in client responses:

* companyId
* createdById
* assignedToUserId
* internalNotes
* applicationNumber, unless explicitly considered safe
* documentUrl if it may expose internal/private files
* any internal-only fields
